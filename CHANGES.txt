── 1. PROJECT SPEC SUMMARY ─────────────────────────────────

This project builds an adversarial, LLM-powered anonymizer that rewrites
sensitive texts to hide specific identities or attributes by neutralizing
semantic clues. The system consists of three competing/cooperating components:
a Defender (anonymizes text), an Attacker (attempts to infer hidden
attributes), and a Utility Judge (ensures the rewrite preserves meaning).
The core challenge is balancing privacy (Attacker can't guess) with utility
(text remains useful).

2.11 NER/REGEX PRE-SCAN LAYER (HYBRID DEFENDER)
    Files: backend/app/ner.py (new)
           backend/app/nodes.py → defender() (modified)
           backend/app/prompts.py → DEFENDER_PROMPT (modified)
           backend/app/state.py → ner_findings field (added)
           backend/pyproject.toml → spacy>=3.7 (added)
    What it does:
      - Deterministic pre-scan using regex and spaCy NER before the LLM
      - Regex detects: EMAIL, PHONE, URL, IP_ADDRESS, SSN, CREDIT_CARD
      - spaCy detects: PERSON, GPE (locations), LOC, ORG
      - Findings are passed to the Defender LLM as advisory context hints
      - Original text is NOT replaced — the LLM decides how to handle each
      - Graceful fallback: if spaCy model is missing, regex-only detection
    Design rationale:
      - Previous implementation already had LLM-based direct identifier
        handling in the Defender prompt ("neutralize any direct identifiers")
      - This new layer adds deterministic regex/spaCy detection as a pre-scan
      - Findings are passed as hints, not permanent replacements, so the LLM
        retains full context and judgment for the rewrite
    Spec requirement satisfied: "a combination of existing NER tooling (like
      REGEX) and LLM based approaches" (Topic6.txt core challenge)
    Setup note:
      - spacy>=3.7 is installed as a dependency
      - For full spaCy NER: python -m spacy download en_core_web_sm
      - If model is missing, app falls back to regex-only (still runs)
    Status: FULLY IMPLEMENTED

── 7. ARCHITECTURE NOTES ────────────────────────────────────

ACTUAL CODE FLOW (graph.py + nodes.py):
    START
      ↓
    defender (nodes.defender)
      - Reads: original_text, attributes_to_hide, feedback, channel, config
      - Writes: current_text, strategy_log, defender_reasoning, iteration++
      ↓
    attacker (nodes.attacker)
      - Reads: current_text, attributes_to_hide, config
      - Writes: attacker_result (dict with guesses)
      ↓
    judge (nodes.judge)
      - Stage 1: Privacy via PRIVACY_PROMPT
        - Reads: attacker_result, original_text, current_text
        - Decides leaked/safe per attribute
      - IF LEAK: short-circuit, write feedback, update best_candidate
      - IF NO LEAK: Stage 2: Utility via UTILITY_PROMPT
        - Scores task_utility, factual_consistency, etc.
        - IF utility OK: verdict="PASS"
        - IF utility LOW: write feedback, update best_candidate
      ↓
    route_after_judge (conditional edge)
      - PASS → finalize
      - NOT PASS + iteration < max_iters → defender (retry)
      - NOT PASS + iteration >= max_iters → finalize (best candidate)
      ↓
    finalize (nodes.finalize)
      - Returns final_text, rounds, verdict
      ↓
    END

README DIAGRAM VS CODE:
    - README shows "leak? → yes → retry" — accurate
    - README shows "utility ok? → no → retry" — accurate
    - README shows "rounds left?" decision — this is implicit in
      route_after_judge, not a separate node
    - README correctly describes two-stage Judge behavior
    - No discrepancies found; README accurately represents the code

STATE MANAGEMENT:
    - AnonState (state.py) uses TypedDict with Annotated[list, operator.add]
      for history (append-only audit trail)
    - best_candidate tracks best attempt across rounds
    - feedback is passed to next Defender iteration

LLM CALLS:
    - 4 potential LLM calls per round: 1 defender + 1 attacker + 1 privacy +
      (0 or 1) utility (skipped on leak)
    - All use structured output via function_calling
    - LLMs are cached per (spec, temperature) pair via lru_cache
